## 4. Multi-option / actionable

Each scenario row routes through `ScenarioIntake` overlay when clicked:
- `onClick={() => onSelectScenario(s)}` → `setIntakeScenario(s)` → `ScenarioIntake` component renders
- `ScenarioIntake` shows scenario context, entity financial context, and routes to `onNav('de', { query, eventId })`
- For engine=true scenarios: `eventId` is set — DecisionEngine can pre-match an event tree (e.g., `eventId: 'retire'`, `'part_time'`, `'sell_property'`, etc.)
- For engine=false scenarios: `query` string is rich and detailed; `eventId: null` — DE handles as open query

The inventory spec (H-OVL-02) requires "A/B/C/D options, each actionable, current-financials-bound, time-projected". The ScenarioIntake → DecisionEngine path delivers multi-option (A/B/C/D comes from DecisionEngineV2 which renders options), but the options are generated by the DE, not hardcoded here. This is the correct architecture for FD-1.

Verdict: **PASS** — multi-option delivered via DE, not hardcoded list ✓

## 5. Finance-bound (uses entity data)

`ScenarioIntake` (`src/components/Home/ScenarioIntake.jsx`) calls `buildEntityContext(entity)` which:
- `netWorth(entity)` → £Xk context
- `calcFQ(entity).total` → Wealth Score
- `calcRisk(entity).total` → Risk Score
- SIPP/pension value calculated from `entity.assets`
- Income from `entity.income`

This context is prepended to the query: `[My financial context: Age: X · Net worth: £Xk · Pension/SIPP: £Xk · Annual income: £Xk · Wealth Score: X/100 · Risk Score: X/100]`

Verdict: **PASS** — scenarios are entity-finance-bound, not generic ✓

## 6. Time-projected

- Engine=true scenarios (retire, part_time, downsize, lump_pension, market_drop, gift): routed to DE via `eventId` — DecisionEngine handles time projection in its option cards.
- Engine=false scenarios (Ask Sonu): routed to DE as a query — DE generates time-projected analysis.
- Inventory requirement (H-OVL-02): "current→target over the stated horizon" — this depends on the DE rendering, not HomeScreen itself.
- The `query` strings in DE_SCENARIOS include time-anchored language: "What if I retired 5 years earlier?" / "What if markets fell 20%? Show ... estimated recovery timeline."

Verdict: **PASS (conditional)** — time projection is the DE's responsibility once the scenario is routed there. HomeScreen correctly passes a time-aware query + entity context. Full pass requires DE audit to confirm options are time-projected.
